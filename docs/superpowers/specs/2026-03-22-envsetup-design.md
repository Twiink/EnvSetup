# EnvSetup 需求文档

## 1. 项目背景

当前团队在开发环境搭建上通常面临以下问题：

- 新成员入场或新机器初始化成本高
- 不同成员的开发环境版本、目录结构和配置方式不一致
- 环境搭建过程依赖个人经验，难以标准化复用
- 安装失败后缺少统一的日志、结果和恢复机制
- 随着技术栈扩展，前端、Java、Python、Maven、Django 等环境能力难以统一管理

因此，本项目计划构建一个基于 **Electron + TypeScript** 的跨平台桌面应用，面向 **Windows** 和 **macOS**，用于为团队成员提供开发环境的一键配置能力。

本产品不是单一的语言环境安装器，而是一个面向团队标准化的开发环境配置平台，其核心模式为：

- 通过 **环境模板** 定义团队标准安装方案
- 通过 **插件机制** 承载不同语言与工具链能力
- 通过 **局部自定义** 在标准化基础上保留必要灵活性

---

## 2. 产品目标

本产品的目标是解决团队开发环境安装分散、标准不一致、搭建成本高的问题，逐步形成可复用、可扩展、可审计的环境安装平台。

核心目标包括：

1. 降低新机器和新成员的开发环境搭建成本
2. 支持团队输出标准化开发环境模板
3. 通过插件机制承载多语言、多工具链环境能力
4. 在模板标准化基础上允许用户进行受控范围内的局部自定义
5. 为后续接入 Java、Python、Maven、Django 等能力建立统一框架

---

## 3. 目标用户

### 3.1 普通开发者

- 选择模板并完成环境安装
- 在模板允许范围内调整路径、版本等参数
- 查看安装过程、结果和失败原因

### 3.2 团队模板维护者

- 维护团队标准环境模板
- 定义插件组合、默认参数和允许覆盖项
- 控制团队安装方案的一致性

### 3.3 插件开发者

- 按统一规范开发环境插件
- 扩展新的语言环境或工具链能力
- 维护插件兼容性与版本演进

## 4. 产品范围

### 4.1 一期范围

一期按“平台基础能力 + 一个完整示例插件”思路建设，不按语言类别切分产品。

一期重点包括：

- 支持 Windows / macOS
- 提供环境模板机制
- 提供 **受信任本地插件机制**
- 支持模板参数局部覆盖
- 支持安装前检查
- 支持安装任务执行与日志反馈
- 支持插件级失败重试
- 支持本地插件导入
- 提供一个完整的前端环境插件示例

### 4.2 一期示例插件：前端环境插件

首个示例插件用于验证平台能力，重点覆盖：

- 在 macOS 上支持“直接安装 Node.js”或“安装 nvm 并安装指定 Node 版本”二选一
- 在 Windows 上支持“直接安装 Node.js”或“安装 nvm-windows 并安装指定 Node 版本”二选一
- 支持 Node 版本选择
- 支持 npm cache 目录配置
- 支持 npm 全局依赖目录（global prefix）配置
- 支持用户态安装目录配置
- 输出安装结果、安装路径、版本信息和关键环境变量结果

说明：

- 一期前端插件仅强制覆盖 **Node / nvm(nvm-windows) + npm** 场景
- pnpm store、Yarn cache 等目录管理不纳入一期强制范围，可作为后续扩展项
- 一期默认优先支持 **用户态安装**，不以系统级安装为默认路径

### 4.3 长期范围

长期将逐步支持：

- 更开放的插件来源与插件治理机制
- Java、Python、Maven、Django 等更多环境插件
- 企业内部插件仓库
- 远程 URL / 插件中心
- 更完善的签名校验、升级、回滚与生态治理能力

### 4.4 非目标

一期明确不包括：

- 公共插件市场
- 云端账号体系
- 所有语言环境同时落地
- 企业级设备批量托管
- 完整插件商业生态
- 复杂权限沙箱体系
- 全面自动回滚系统
- 远程插件下载与在线升级

## 5. 系统架构与职责边界

### 5.1 分层架构

系统按以下职责分层：

1. **渲染进程（UI 层）**
   - 负责模板选择、参数填写、安装进度与结果展示
   - 不直接执行系统安装命令
   - 不直接访问敏感系统能力

2. **预加载层（Bridge）**
   - 负责向 UI 暴露受控 API
   - 负责参数校验、数据传输、事件订阅桥接

3. **主进程（Orchestrator）**
   - 负责任务编排、插件加载、权限确认、日志汇总、状态持久化
   - 负责调用受控插件执行器

4. **插件执行器（Plugin Runner）**
   - 负责执行插件生命周期钩子
   - 负责收集步骤日志与结构化结果
   - 不负责界面展示

### 5.2 职责边界

- **模板** 仅负责声明式配置，不包含任意执行逻辑
- **插件** 负责具体环境能力实现，不直接修改 UI
- **主程序** 负责插件装载、任务编排、风险提示、状态与日志持久化
- **UI** 只做展示和用户交互，不直接持有系统级权限

### 5.3 执行模型

一期采用受控执行模型：

- 插件通过统一生命周期接口接入
- 主程序按模板和依赖顺序触发插件执行
- 插件执行结果必须通过统一协议返回给主程序
- 一期只要求支持 **受信任本地插件**，不要求构建完整开放生态执行沙箱

## 6. 兼容性矩阵与平台边界

### 6.1 一期支持矩阵

一期明确支持：

- **Windows**：x64
- **macOS**：Apple Silicon、Intel

### 6.2 一期 shell 与环境变量边界

一期至少考虑：

- macOS：zsh、bash
- Windows：PowerShell

说明：

- 一期不要求覆盖 fish、cmd 等全部 shell
- 一期默认以 **当前用户级环境变量配置** 为主，不以系统级环境变量修改为默认路径

### 6.3 当前不纳入一期承诺的范围

- Linux 支持
- 全量 shell 兼容
- 离线安装全覆盖
- 企业设备受限策略下的全部兼容场景

## 7. 核心概念模型

### 7.1 环境模板（Environment Template）

环境模板是用户安装环境时的主入口，用于定义一套可复用的标准开发环境方案。

模板应至少包含以下内容：

- 模板基础信息
  - 名称
  - 唯一标识
  - 版本
  - 适用平台
  - 描述
- 所包含的插件列表
- 每个插件的默认配置
- 允许用户覆盖的配置项列表
- 插件执行顺序
- 模板级前置检查与安装后检查
- 模板级安装说明与风险提示

模板的核心作用是由团队先定义标准环境，再允许用户在受控范围内进行局部调整。

### 7.2 环境插件（Environment Plugin）

插件是环境能力的最小交付单元，用于承载某项环境安装或配置能力。例如：Node 插件、nvm 插件、Python 插件、Java 插件、Maven 插件、Django 插件。

每个插件应定义：

- 插件元数据
  - 名称
  - 唯一标识
  - 版本
  - 作者或来源
  - 支持平台
  - 兼容的主程序版本范围
- 插件能力描述
  - 安装目标
  - 输入参数
  - 会修改的系统配置
  - 对其他插件的依赖关系
- 插件生命周期钩子
  - 环境检查
  - 安装前准备
  - 执行安装
  - 安装后校验
  - 卸载或回滚（如支持）
- 插件输出
  - 安装结果
  - 日志
  - 生成的环境变量
  - 向后续插件提供的上下文信息

插件需要具备可独立安装、独立校验、独立失败的能力，并通过统一接口接入主程序。

### 7.3 安装任务（Install Task / Execution Plan）

当用户选定模板并确认参数后，系统生成一次安装任务。

安装任务至少应包含：

- 任务 ID
- 来源模板
- 使用的插件及版本快照
- 用户最终确认的参数
- 当前执行阶段
- 每个插件的执行状态
- 成功 / 失败结果
- 日志与错误信息
- 开始时间、结束时间、耗时

安装任务负责将“模板 + 插件 + 用户输入”转化为一条可跟踪、可展示、可恢复的环境配置流程。

### 7.4 用户覆盖配置（User Overrides）

产品采用“模板优先 + 局部自定义”模式，因此需要明确用户允许修改的边界。

用户覆盖配置可能包括：

- 安装路径
- 缓存目录
- 全局依赖目录
- 指定版本
- 是否启用某个可选插件
- 插件内部的少量高级参数

模板应明确：

- 哪些配置不可修改
- 哪些配置允许修改
- 哪些配置仅允许在合法范围内修改
- 哪些配置会影响后续插件依赖

### 7.5 执行结果（Execution Result）

一次安装完成后，系统需要输出统一结果模型，用于界面展示、问题排查和后续重试。

结果至少应包含：

- 总体成功 / 部分成功 / 失败
- 每个插件的执行结果
- 失败步骤及原因
- 建议修复动作
- 实际写入的路径与配置
- 已安装的软件版本
- 关键环境变量变更
- 可导出的安装报告

## 8. 模板与插件格式最小规范

### 8.1 模板文件最小规范

一期模板采用 **声明式文件格式**，具体编码格式可在技术方案阶段确定为 JSON 或 YAML，但在需求层需满足以下最小字段：

- `id`
- `name`
- `version`
- `platforms`
- `description`
- `plugins`
- `defaults`
- `overrides`
- `checks`

约束要求：

- `id` 全局唯一
- `version` 使用 semver
- `plugins` 使用 `pluginId + version` 方式引用
- 模板本身不允许包含任意可执行脚本

### 8.2 插件清单最小规范

一期插件包需包含最小 manifest，至少包含：

- `id`
- `name`
- `version`
- `mainAppVersion`
- `platforms`
- `permissions`
- `parameters`
- `dependencies`
- `entry`

约束要求：

- `id` 全局唯一
- `version` 使用 semver
- `permissions` 为声明式列表
- `entry` 指向插件生命周期入口
- 插件包必须能被主程序做静态校验

### 8.3 参数类型最小集合

一期参数类型至少支持：

- string
- boolean
- enum
- path
- version
- number

### 8.4 生命周期接口最小集合

一期插件生命周期最小集合定义为：

- `check`
- `prepare`
- `install`
- `verify`

`rollback` 为可选能力，不要求所有插件必须实现。

### 8.5 参数寻址与覆盖规则

一期参数覆盖采用统一寻址方式：

- 模板对插件参数的引用方式为 `pluginId.parameterKey`
- `defaults` 用于声明模板给定的默认值
- `overrides` 用于声明该参数是否允许用户修改，以及修改约束

`overrides` 最小需支持以下属性：

- `editable`: 是否允许修改
- `required`: 是否必填
- `enum`: 可选值集合
- `range`: 数值范围
- `pattern`: 字符串或路径校验规则
- `affects`: 该字段变化会影响的插件列表
- `dependsOn`: 该字段显示或可编辑依赖的前置条件

### 8.6 互斥、依赖与校验规则

一期采用如下最小规则：

- **插件依赖关系** 由插件 manifest 声明
- **模板组合限制** 可由模板补充声明
- **参数层互斥或分支选择** 由模板 `overrides` 表达
- 模板级 `checks` 先执行，插件级 `check` 后执行
- 预检结果由主程序聚合为统一输出

例如：

- “Node 直装”与“nvm / nvm-windows 管理”属于模板层参数分支互斥
- Django 依赖 Python 属于插件依赖
- Maven 依赖 Java 属于插件依赖

## 9. 插件信任模型与执行边界

### 9.1 一期“受信任本地插件”定义

一期“受信任本地插件”指：

- 仅允许用户从本地目录或本地压缩包手动导入
- 默认视为由团队内部维护或由用户自行确认来源
- 不支持一期内从远程地址自动下载、自动更新或动态拉取执行代码

### 9.2 导入时主程序必须校验的内容

主程序在导入插件时必须至少校验：

- manifest 是否完整且字段合法
- 插件 ID、版本号是否合法
- 主程序版本兼容范围是否满足
- 平台声明是否匹配当前系统
- `entry` 是否存在
- 插件包结构是否符合预期

可选扩展位：

- checksum
- 签名信息

### 9.3 权限声明的效力边界

一期中，插件 `permissions` 的效力定义为：

- **必须声明** 可能涉及的高风险操作类型
- 主程序必须在执行前进行展示与确认
- 一期不要求实现“未声明权限即从系统层强制阻止”的完整沙箱
- 但主程序应对明显越界行为保留后续升级空间

### 9.4 一期执行边界

一期执行边界收敛为：

- 插件通过统一生命周期接口接入
- 主程序统一负责任务编排、参数注入、日志收集与结果聚合
- 一期可以允许插件入口执行受控 Node 侧逻辑
- 但插件不得直接操作渲染进程 UI，不得绕过主程序直接接入产品状态管理

## 10. 权限与提权策略

### 10.1 一期默认策略

一期默认采用 **用户态安装优先** 策略：

- 默认安装到当前用户可写目录
- 默认修改当前用户级环境变量或用户级 shell 配置
- 非必要场景不执行系统级安装

### 10.2 提权原则

当插件涉及以下操作时，需要显式提示并走提权流程：

- 写入系统级目录
- 修改系统级环境变量
- 调用需要管理员权限的系统安装行为

### 10.3 一期提权边界

一期对提权能力的要求是：

- 允许识别“需要提权”的操作
- 允许用户确认或拒绝
- 用户拒绝后任务可失败并给出明确提示
- 不要求一期实现复杂的细粒度权限沙箱

### 10.4 日志脱敏要求

日志和报告中不得直接暴露敏感信息，包括但不限于：

- 密钥、token、密码
- 完整凭证参数
- 可识别的隐私信息

## 11. 任务状态机、日志与持久化

### 11.1 任务状态机

一期安装任务至少包含以下状态：

- `draft`
- `prechecking`
- `ready`
- `running`
- `failed`
- `partially_succeeded`
- `succeeded`
- `cancelled`

### 11.2 状态迁移原则

- 用户确认配置后，任务从 `draft` 进入 `prechecking`
- 预检通过后进入 `ready`
- 开始安装后进入 `running`
- 全部插件成功后进入 `succeeded`
- 存在失败插件且存在已成功插件时进入 `partially_succeeded`
- 发生失败且无成功结果时进入 `failed`
- 用户主动中止后进入 `cancelled`

### 11.3 插件最小执行状态

为支持插件级恢复，一期每个插件至少需要记录：

- `not_started`
- `running`
- `installed_unverified`
- `verified_success`
- `failed`
- `needs_rerun`

### 11.4 日志持久化

每个安装任务应生成唯一任务记录，至少持久化：

- 任务基础信息
- 模板快照
- 插件版本快照
- 用户确认参数
- 结构化状态流转
- 插件执行日志
- 插件输出上下文快照
- 最终结果摘要

### 11.5 恢复判定规则

一期恢复能力收敛为 **插件级重试**，并采用以下最小判定规则：

- 若插件参数变化，则该插件状态置为 `needs_rerun`
- 若插件版本变化，则该插件状态置为 `needs_rerun`
- 若其依赖插件输出上下文变化，则该插件状态置为 `needs_rerun`
- 若上次状态为 `verified_success` 且本次 `check/verify` 仍通过，可跳过执行
- 若上次状态为 `installed_unverified`，则至少需要重新执行 `verify`，必要时重新执行插件
- 若用户主动取消，则已完成插件保持原状态，未完成插件标记为未完成，不承诺自动回滚

### 11.6 成功跳过与强制重跑规则

- 仅当插件版本、参数、依赖上下文均未变化，且插件自检通过时，允许跳过
- 任一条件不满足时，主程序必须将插件标记为需要重跑
- 模板整体变化是否影响插件，由解析后的参数与依赖上下文变化决定，而不是仅看模板版本号

## 12. 错误模型

一期至少应支持以下错误分类：

- 平台不支持
- 权限不足
- 参数非法
- 路径不可写
- 网络失败
- 下载校验失败
- 环境冲突
- 插件包不合法
- 插件依赖缺失
- 插件执行异常
- 安装后校验失败
- 用户取消

错误输出至少包含：

- 错误分类
- 错误描述
- 发生插件
- 发生步骤
- 建议修复动作

## 13. 核心业务流程

### 13.1 模板选择

用户进入应用后，应能够选择模板并查看以下信息：

- 模板名称与用途
- 适用平台
- 所包含的插件
- 推荐使用场景
- 模板版本
- 模板维护者
- 是否为团队推荐模板

在后续版本中，模板来源可区分为：

- 官方模板
- 团队模板
- 本地导入模板

### 13.2 参数覆盖

用户选定模板后进入局部自定义阶段，仅能修改模板允许覆盖的配置项，例如：

- 安装路径
- 缓存目录
- 全局依赖目录
- Node 管理方式（直接安装 Node / 使用 nvm）
- 指定软件版本
- 是否启用某个可选插件

该阶段需要强调：

- 清楚标识默认值
- 清楚标识模板值与用户修改值
- 清楚标识某项修改会影响哪些插件
- 对非法输入进行即时校验
- 对高风险改动给予提示

### 13.3 安装前检查

在执行安装前，系统需要提供预检结果，尽量提前暴露风险。

预检至少覆盖：

- 当前操作系统是否支持
- 当前架构是否支持
- 目标目录是否可写
- 是否存在已有环境冲突
- 是否缺少必要权限
- 网络条件是否满足下载要求
- 插件依赖是否完整
- 模板与插件版本是否兼容

预检结果分为：

- `pass`：可直接继续
- `warn`：有风险但可继续
- `block`：必须修复后才能继续

### 13.4 安装执行

安装执行阶段应以任务视角展示，而不是仅提供一个总进度条。

建议展示：

- 当前执行的插件
- 当前插件的具体步骤
- 已完成 / 进行中 / 失败状态
- 总体进度
- 关键日志
- 当前写入路径或下载动作
- 是否允许取消

### 13.5 结果反馈

安装完成后，系统应输出结构化结果。

成功场景需展示：

- 安装成功的插件列表
- 实际安装版本
- 实际路径
- 已配置的缓存 / 全局依赖目录
- 关键环境变量结果
- 后续建议操作

失败场景需展示：

- 哪个插件失败
- 失败在哪一步
- 失败原因说明
- 是否可重试
- 是否建议跳过重新开始
- 是否支持回滚
- 日志查看入口

### 13.6 失败恢复与重试

一期恢复能力定义如下：

- 整体任务失败后可重新执行
- 某个插件失败后可进行插件级重试
- 已成功插件是否跳过，由持久化状态与插件自检联合决定
- 高风险插件可声明支持回滚，但不是一期统一强制要求

### 13.7 取消行为

一期取消行为定义如下：

- 允许在 `running` 状态下发起取消
- 取消后任务状态记为 `cancelled`
- 取消不承诺自动回滚已完成操作
- 取消后需明确提示哪些插件已成功执行、哪些步骤可能已对系统产生影响

## 14. 插件系统要求

### 14.1 一期插件机制

一期插件机制的目标是验证平台可行性，而不是一次性完成开放生态建设。

一期要求：

- 仅支持 **受信任本地插件**
- 支持本地目录或本地压缩包导入
- 主程序可识别、校验并注册插件
- 模板可引用指定插件和版本
- 插件执行结果可由主程序统一收集与展示

### 14.2 长期开放机制

长期方向上，系统将逐步支持：

- 官方插件与第三方插件统一接入
- 插件独立打包与分发
- 企业仓库与远程来源
- 更严格的签名、校验与治理机制

### 14.3 插件依赖关系

插件之间可能存在依赖关系，例如：

- Django 插件依赖 Python 插件
- Maven 插件依赖 Java 插件
- 某些前端工具链插件依赖 Node 或 nvm 插件

因此插件模型需要支持：

- 显式声明依赖插件
- 声明依赖版本范围
- 声明互斥关系
- 声明推荐组合
- 安装顺序自动推导或模板显式指定

### 14.4 插件权限边界

由于插件会执行系统级安装动作，因此需求中必须定义权限边界。

插件至少应声明可能涉及的操作类型，例如：

- 下载文件
- 解压与写入本地目录
- 修改环境变量
- 修改 shell 配置
- 执行安装脚本
- 写入缓存目录
- 写入全局工具目录
- 调用系统包管理器

主程序应在安装前向用户展示关键风险操作，并在执行时记录实际发生的关键变更。

### 14.5 插件来源管理

插件来源分阶段支持：

- **一期**：本地目录、本地压缩包导入
- **二期**：企业内部插件仓库
- **三期**：远程 URL / 插件中心

统一来源抽象至少包括：

- 来源类型
- 来源地址
- 下载或导入状态
- 来源可信度标记
- 签名或校验字段（后续扩展位）

### 14.6 版本兼容与升级

至少需要考虑：

- 插件版本与主程序版本兼容关系
- 模板绑定的插件版本
- 安装任务执行时冻结模板与插件快照
- 插件升级后是否影响已有模板
- 同一模板在不同机器上的可复现性

### 14.7 插件失效与异常处理

插件可能出现以下异常情况：

- 插件包损坏
- 元数据不合法
- 缺少依赖
- 不支持当前平台
- 与当前模板冲突
- 执行中失败
- 已安装但校验不通过

主程序应尽量在安装前拦截问题，在安装中清晰暴露错误，在安装后给出可重试或修复建议。

## 15. 前端环境插件 MVP 约束

### 15.1 版本选择策略

一期 Node 版本选择至少支持以下一种稳定方案，具体在技术实现阶段二选一：

- 内置受支持版本列表
- 启动时从受控远程源拉取版本列表并缓存

需求层要求：

- 用户不能输入任意非法版本字符串
- 版本选择结果必须可被记录到安装任务快照中

### 15.2 下载来源策略

一期下载来源策略要求：

- 默认使用官方或团队指定镜像源
- 下载源必须在 UI 或日志中可追踪
- 后续可扩展镜像源配置，但不是一期强制能力

### 15.3 校验策略

一期对下载产物的最小要求：

- 至少要有下载失败与文件不可用的错误识别
- checksum 校验建议支持，但如一期成本过高，可作为“应支持”能力在技术方案中明确取舍

### 15.4 已有环境冲突策略

当前机器若已存在 Node、nvm 或 nvm-windows，一期默认策略如下：

- 预检必须识别并提示已有环境
- 默认不静默覆盖已有环境
- 允许继续安装、终止安装或按后续技术方案定义的“复用已有环境”策略执行
- 最终采用何种处理方式，必须在技术方案中对 macOS 与 Windows 分别明确

### 15.5 成功验收标准

前端环境插件安装成功的最小判定标准为：

- 插件安装步骤执行完成
- `verify` 能确认目标版本存在
- 目标目录、npm cache、global prefix 配置结果可读取
- 至少满足“新开支持的一期终端后可用”的标准

## 16. 平台生效策略

### 16.1 macOS 生效策略

一期 macOS 的最小口径：

- 优先支持 zsh，其次兼容 bash
- shell 初始化片段由主程序或插件按统一规则写入用户级 profile
- 对 nvm 初始化片段必须避免重复写入
- 成功标准以“新开终端后可用”为准

### 16.2 Windows 生效策略

一期 Windows 的最小口径：

- 以用户级环境变量为主
- 以 PowerShell 为首要验证终端
- 对 Node 直装与 nvm-windows 两种模式，均需明确安装后 PATH 生效路径
- 成功标准以“新开 PowerShell 后可用”为准

### 16.3 当前任务内可用与新终端可用

一期以 **新开支持终端后可用** 作为验收标准。当前安装任务执行过程中，如主程序需要使用新环境，可通过显式路径或任务上下文传递，不以“当前终端自动刷新生效”为强制要求。

## 17. 模板维护者与插件开发者工作流

### 17.1 模板维护者最小工作流

一期至少支持模板维护者完成：

- 编写模板声明文件
- 选择插件与版本
- 设置默认值
- 声明允许覆盖项
- 本地导入并验证模板

### 17.2 插件开发者最小工作流

一期至少支持插件开发者完成：

- 编写插件 manifest
- 实现生命周期入口
- 声明参数、权限和依赖
- 打包为本地插件包
- 在主程序中导入并验证

## 18. 非功能需求

### 18.1 跨平台要求（必须）

产品必须支持 Windows 与 macOS，并将跨平台差异作为核心要求处理。

至少需要分别考虑：

- 安装路径规则
- 环境变量写入方式
- shell 配置差异
- 权限模型差异
- 默认缓存目录策略
- 默认全局依赖目录策略
- 系统包管理器或安装方式差异

### 18.2 可扩展性（必须）

系统必须支持未来逐步扩展到 Java、Python、Maven、Django 等环境能力。

可扩展性至少体现在：

- 新插件接入不应改动主流程架构
- 模板可自由组合插件
- 插件之间支持依赖与上下文传递
- 插件来源可从本地逐步扩展到仓库与远程中心
- UI 可承载越来越多插件类型而不失控

### 18.3 安全性（必须）

由于系统会执行安装、写入目录、修改配置、调用外部安装程序，安全性必须作为核心要求。

至少要覆盖：

- 插件来源标识与可信度区分
- 插件声明自身权限需求
- 主程序在执行前向用户展示关键高风险操作
- 执行过程记录关键变更
- 对第三方插件预留签名 / 校验扩展能力
- 对危险路径写入、脚本执行、环境变量覆盖进行提示

### 18.4 可观测性（必须）

环境安装类产品需要较强的可观测性。

至少要求：

- 任务级日志
- 插件级日志
- 步骤级状态
- 错误码或错误分类
- 安装结果报告导出
- 关键操作时间点记录

### 18.5 稳定性与可恢复性（必须）

安装过程可能受网络、权限、磁盘、已有环境冲突等因素影响，因此需要具备较强稳健性。

至少要求：

- 安装中断后状态可识别
- 重试机制清晰
- 插件级恢复边界清晰
- 对部分成功状态可识别
- 对失败插件提供明确恢复建议

### 18.6 性能与体验（应该）

- 应用启动速度可接受
- 大型插件安装时界面不应卡死
- 日志刷新与进度展示保持流畅
- 下载、解压、安装执行需异步处理
- 长时间任务中持续给用户反馈

### 18.7 可维护性（必须）

为了支持未来插件数量增长，需要明确边界：

- 主程序负责模板、任务编排、日志、权限提示、插件管理
- 插件负责具体环境能力实现
- 模板负责标准方案编排
- 插件元数据、配置模型、执行协议需要统一

### 18.8 本地数据与隐私（应该）

- 任务状态和日志默认保存在本地
- 未经明确设计前，不上传遥测数据
- 导出报告前应进行敏感信息脱敏

## 19. MVP 定义与验收标准

### 19.1 MVP 必须实现

- Electron + TypeScript 桌面应用基础框架
- Windows / macOS 支持
- 环境模板选择能力
- 模板参数覆盖能力
- 插件注册与执行机制
- 本地插件导入能力
- 安装前检查
- 任务执行状态展示
- 日志与安装结果展示
- 插件级失败重试能力
- 一个完整的前端环境插件示例

### 19.2 前端环境插件 MVP 要求

- 安装 Node.js
- 或安装 / 管理 nvm（macOS）与 nvm-windows（Windows）
- Node 与 nvm 方案二选一
- 配置 npm cache 目录
- 配置 npm 全局依赖目录
- 支持用户态安装路径配置
- 支持版本选择
- 输出安装结果与环境信息

### 19.3 MVP 明确不做

- 公共插件市场
- 云端账号体系
- 插件在线评分 / 审核
- 企业级设备批量管控
- 所有语言插件同时落地
- 复杂自动回滚体系
- 过于复杂的权限沙箱体系
- 通用步骤级恢复引擎
- 远程插件下载与在线升级

### 19.4 MVP 验收标准

以下条件满足时，可判定 MVP 达标：

1. 用户能够在 UI 中选择一个模板并查看其插件组成
2. 用户能够修改模板允许覆盖的字段，且非法输入会被阻止
3. 系统能够执行预检并输出 `pass / warn / block` 三类结果
4. 系统能够导入并校验一个本地插件包
5. 系统能够按模板顺序执行插件，并展示任务状态变化
6. 任务失败后，用户能够重新发起插件级重试
7. 任务完成后，系统能够展示安装结果、版本、路径和关键日志
8. 前端环境插件能够在支持的平台上完成 Node 或 nvm(nvm-windows) 路径的安装流程

## 21. 风险与待确认问题

### 21.1 主要风险

1. **跨平台差异风险**
   Windows 与 macOS 在路径、环境变量、权限和安装机制上差异明显。

2. **插件安全风险**
   开放插件机制天然带来脚本执行、路径写入、来源可信度问题。

3. **环境冲突风险**
   用户机器可能已安装 Node、Python、Java 或其他工具链，存在兼容和覆盖问题。

4. **可恢复性风险**
   安装中断后若无法识别部分成功状态，可能导致机器环境不一致。

5. **需求膨胀风险**
   如果一期同时追求平台化、开放生态和多语言全覆盖，项目实施风险会显著上升。

### 21.2 待后续进一步设计的问题

- 插件包标准格式的精确定义
- 插件生命周期接口的具体 TypeScript 类型
- 模板文件编码格式最终选择（JSON / YAML）
- 插件执行隔离程度
- 提权交互的具体技术实现
- 日志文件存储位置与清理策略
- 回滚能力的最小统一协议
- 代理、镜像源、离线安装等网络策略

## 22. 总结

本项目定位为一个基于 **Electron + TypeScript** 的跨平台开发环境配置平台，面向 Windows 与 macOS，采用“环境模板 + 插件机制”的产品架构，支持团队标准化环境方案下发，并允许用户在受控范围内进行局部自定义。

一期以 **受信任本地插件机制** 和 **前端环境插件** 为落地示例，目标是先验证平台架构、安装流程、日志与恢复模型的可行性，再逐步扩展到 Java、Python、Maven、Django 等多种开发环境能力，并最终演进为更开放的插件生态。
